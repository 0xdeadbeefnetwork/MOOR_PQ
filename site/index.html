My Own Onion Router

Anonymous overlay network with 3-hop onion routing, post-quantum hybrid cryptography on every hop, every link, every signature, hidden services, and traffic analysis resistance. ~34,000 lines of C across 40 source files. ~3,600 lines of vendored NIST PQ reference implementations. Three dependencies. No OpenSSL.

Features

What ships in v0.7.0

Cryptography
🛡

Post-Quantum Hybrid Cryptography

The quantum threat isn't the encryption — ChaCha20 with 256-bit keys already has 128-bit quantum security. The threat is the key exchange. A quantum computer running Shor's algorithm can break X25519 Diffie-Hellman and recover the symmetric keys from recorded traffic.

MOOR prevents this. At every hop, the ChaCha20 session key is derived from both an X25519 DH shared secret and a Kyber768 KEM shared secret, combined via BLAKE2b. An attacker must break both X25519 and Kyber768 (NIST Level 3) to recover any key. Quantum breaks X25519 but not Kyber768 — so the key survives, the encryption survives, the traffic stays sealed.

This isn't optional. There is no flag to disable it. Every circuit hop, every link handshake, every consensus signature uses hybrid classical + post-quantum cryptography. Traffic recorded today cannot be decrypted by a quantum computer tomorrow.

🔏

ML-DSA-65 Consensus Signatures

Directory Authority signs consensus with Ed25519 + ML-DSA-65 (FIPS 204 Dilithium3) hybrid. Backward-compatible wire format with PQ magic bytes. Post-quantum authenticity for network state.

⚡

Noise_IK Link Protocol

1-RTT handshake. No certificate authority. The relay's Ed25519 identity key is the trust anchor. ChaCha20-Poly1305 + BLAKE2b. ~300 lines of implementation vs. hundreds of thousands for TLS.

Routing & Privacy
🌐

3-Hop Onion Routing

Your traffic goes through 3 relays: Guard (sees your IP, not your destination), Middle (sees nothing), Exit (sees your destination, not your IP). No single relay can link you to what you're accessing. Relays are selected weighted by bandwidth, your guard stays the same for months to limit exposure, and no two relays in a circuit share the same network or country.

🎭

Hidden Services

Host a website or service that's only reachable through the MOOR network using a .moor address. Traffic between client and hidden service is end-to-end encrypted — even the relay that connects them sees only ciphertext. Service descriptors are stored encrypted across multiple relays. Vanguard relays protect your service from guard discovery attacks. Proof-of-work prevents denial-of-service flooding. OnionBalance lets you load-balance across multiple backend servers.

🕵

PIR for Hidden Service Lookups

When you visit a .moor hidden service, the relay storing the service descriptor could learn which service you're looking for. MOOR prevents this: your lookup query is split across 2 relays, each seeing only random noise. Neither relay can determine which .moor address you requested. This eliminates the biggest metadata leak in hidden service lookups.

🔒

End-to-End HS Encryption

ChaCha20-Poly1305 AEAD on every RELAY_DATA cell between client and hidden service. Post-quantum hybrid key exchange (X25519 + Kyber768) at rendezvous — the X25519 handshake completes first, then Kyber768 KEM immediately upgrades the session to PQ hybrid. The RP relay — the node that bridges both sides — sees only ciphertext. Even a compromised rendezvous point cannot read your traffic.

Traffic Analysis Resistance
📡

Traffic Analysis Resistance

All data travels in fixed 514-byte cells — a 10-byte message and a 498-byte page look identical on the wire. Even when you're not browsing, MOOR sends padding cells at a constant rate so an observer can't tell if you're active or idle. Each circuit gets a randomized padding pattern. Multi-path circuits can split your traffic across different routes simultaneously. Pluggable transports (Scramble, Shade, Mirage) disguise MOOR traffic as normal HTTPS.

🔀

Poisson Relay Mixing

Relays can hold each cell for a random delay before forwarding it. Real traffic and cover traffic share the same delay pool — an observer watching a relay's input and output can't correlate which incoming cell became which outgoing cell. The delay is configurable per-relay.

🧩

WTF-PAD Adaptive Padding

Intelligent padding that adapts to your traffic pattern. Three presets: web mimics normal browsing bursts, stream produces constant-rate flow, generic provides balanced coverage. Each circuit gets its own randomized padding schedule so no two circuits look alike, even from the same client.

Network & Performance
🚀

Vegas Congestion Control

Smart congestion control that measures round-trip times and adjusts sending speed per-circuit, so one slow path doesn't kill your other connections. Relays can pause and resume individual streams when overloaded. Flow control cells are authenticated to prevent spoofing.

⚡

Concurrent Circuit Building

MOOR pre-builds a pool of 8 ready circuits in the background. When you open a new connection, you get an instant circuit instead of waiting for a 3-hop handshake. Pages from the same site share a circuit. Real web browsing works at usable speeds through 3 encrypted hops.

🔐

Encrypted DNS (DoT)

Exit relays resolve hostnames via DNS-over-TLS to Cloudflare, Quad9, and Google. Minimal TLS 1.3 client using only libsodium (X25519, ChaCha20-Poly1305, SHA-256). SPKI pinning for server authentication. No plaintext DNS fallback — if all providers fail, the stream is refused. DNS rebinding protection blocks internal IP resolution.

Infrastructure
📦

Single Binary, Minimal Dependencies

Client, relay, DA, hidden service, BridgeDB, OnionBalance — all one binary. Dependencies: libsodium, zlib, pthreads. No OpenSSL. No runtime. Auditable by one person.

🗳

Multi-DA Consensus Voting

Multiple independent directory authorities vote on the list of relays in the network. Each DA measures relay bandwidth, assigns performance flags (Fast, Stable, Guard), and signs the consensus with both classical and post-quantum signatures. All DAs must agree — a single compromised DA cannot forge the network directory.

🌉

Censorship Circumvention

Bridge relays not listed in public consensus. BridgeDB distributes bridge addresses via HTTPS. Bridge Authority for trusted bridge management. Pluggable transports disguise MOOR traffic.

🧠

Argon2id Memory-Hard PoW

To join the network, a relay must solve a memory-hard proof-of-work puzzle (Argon2id, 256KB per hash). This prevents attackers from cheaply flooding the network with thousands of fake relays. GPUs get almost no advantage over CPUs. Hidden services can also require PoW from connecting clients to prevent denial-of-service.

Architecture

Protocol overview

Fixed-Size Cells

Every cell is exactly 514 bytes plaintext (4-byte circuit ID, 1-byte command, 509-byte payload). On the wire: 2-byte length prefix + 514-byte ciphertext + 16-byte Poly1305 MAC = 532 bytes. Relay cells carry an 11-byte header leaving 498 bytes for data. No variable-length records. No size-based traffic analysis.

Layered Hybrid Cryptography

At each hop, the client and relay perform a hybrid key exchange: X25519 DH produces a classical shared secret, Kyber768 KEM produces a post-quantum shared secret. Both are combined via BLAKE2b to derive the ChaCha20 session key. An attacker must break both algorithms to recover any hop's key. Each cell is then encrypted with 3 layers of ChaCha20 (one per hop) and sealed in a ChaCha20-Poly1305 AEAD link frame. Consensus is dual-signed with Ed25519 + ML-DSA-65.

Event-Driven I/O

epoll on Linux, select on Windows, poll fallback elsewhere. Single-threaded event loop with a pthread crypto worker pool and a background circuit builder thread. Expensive Kyber operations are offloaded to workers, which signal completion via pipe fd back to the event loop.

Hidden Service Rendezvous

Client builds a 3-hop circuit to a rendezvous point. HS builds a 3-hop circuit to the same point. Result: a 6-hop tunnel with post-quantum hybrid end-to-end encryption (X25519 + Kyber768 → ChaCha20-Poly1305) — the rendezvous relay sees only ciphertext. Descriptor lookups use 2-server XOR PIR so no relay learns which .moor address was requested. Vanguards (L2/L3 restricted middle hops) protect the HS guard from discovery attacks.

Adversary Capability Matrix

What can each adversary learn from their vantage point? No single position can link client identity to destination.

No readable content

Each cell is wrapped in 3 layers of ChaCha20 stream cipher (one per hop) and then sealed in a ChaCha20-Poly1305 AEAD link frame. The onion layers use keyed ChaCha20 with per-hop nonces — each relay strips one layer. A running BLAKE2b digest authenticates cells at the correct hop. The link-layer AEAD prevents tampering in transit. An observer between any two relays sees only authenticated ciphertext.

No size fingerprinting

Every cell is exactly 514 bytes plaintext. On the wire: 2-byte length prefix + 514 bytes ciphertext + 16-byte Poly1305 MAC = 532 bytes. A 10-byte ping and a 498-byte data transfer produce identical packet sizes. No variable-length records, ever.

Statistically indistinguishable from random

The link layer uses ChaCha20-Poly1305 IETF AEAD (libsodium crypto_aead_chacha20poly1305_ietf) with a 64-bit counter nonce. The only structured bytes are the 2-byte length prefix, which is constant (always 530). No protocol headers, no version fields, no magic bytes. The ciphertext is IND-CPA secure — computationally indistinguishable from random.

Quick Start

Build and run a network

Quick install 
# Install and run (Linux / macOS)
$ curl -sL https://raw.githubusercontent.com/0xdeadbeefnetwork/MOOR_PQ/main/setup.sh | sudo bash
$ moor                    # Browse anonymously (SOCKS5 on :9050)
Build from source 
# Install dependencies (Ubuntu/Debian)
$ sudo apt install build-essential libsodium-dev zlib1g-dev pkg-config

# Build
$ ./configure
$ make -j$(nproc)
$ make check            # run test suite
$ sudo make install
Run a relay (moorrc) 
# Create moorrc config file:
# All relays start as middle relays. The DA assigns flags:
#   Running  — immediately (DA can reach you)
#   Fast     — bandwidth above 12.5th percentile
#   Stable   — uptime above median
#   Guard    — Fast + Stable + bandwidth + 8+ days uptime
#   HSDir    — Fast + Stable + 96+ hours uptime
# On small networks (<20 relays), Guard/Exit granted immediately.

# --- Guard relay moorrc ---
Mode relay
BindAddress 0.0.0.0
ORPort 9001
AdvertiseAddress YOUR_PUBLIC_IP
Guard 1
Nickname MyGuard
DataDirectory /var/lib/moor
ContactInfo you@example.com
Verbose 1

# --- Middle relay moorrc ---
Mode relay
BindAddress 0.0.0.0
ORPort 9001
AdvertiseAddress YOUR_PUBLIC_IP
Nickname MyMiddle
DataDirectory /var/lib/moor
ContactInfo you@example.com
Verbose 1

# --- Exit relay moorrc ---
Mode relay
BindAddress 0.0.0.0
ORPort 9001
AdvertiseAddress YOUR_PUBLIC_IP
ExitRelay 1
Nickname MyExit
DataDirectory /var/lib/moor
ContactInfo you@example.com
Verbose 1

# Run:
$ moor -f moorrc

# NAT: BindAddress = local (0.0.0.0), AdvertiseAddress = public IP
# Port 9001/tcp must be open inbound through your firewall
# All relays must run the SAME version (old binaries fail handshakes)
Browse through MOOR 
# Client moorrc:
SocksPort 9050

# Run:
$ moor -f moorrc

# SOCKS5 proxy on 127.0.0.1:9050. Test with curl:
$ curl -x socks5h://127.0.0.1:9050 https://check.torproject.org/api/ip
{"IsTor":true,"IP":"86.54.28.49"}

$ curl -x socks5h://127.0.0.1:9050 https://example.com
<!doctype html>...

# Hidden service:
$ curl -x socks5h://127.0.0.1:9050 http://<address>.moor/
Browser setup 
# Firefox:
#   Settings → Network Settings → Manual proxy
#   SOCKS Host: 127.0.0.1    Port: 9050    SOCKS v5
#   Check “Proxy DNS when using SOCKS v5”

# Chromium / Brave:
$ chromium --proxy-server="socks5://127.0.0.1:9050"

Bridges

Censorship circumvention

In censored environments where connections to known MOOR relays are blocked, bridge relays provide an alternative entry point. Bridges are not listed in the public consensus, making them harder for censors to discover and block. Combined with pluggable transports, bridge connections are indistinguishable from normal HTTPS traffic.

1. Run a Bridge Relay

A bridge is a relay not listed in the public consensus. Create a moorrc:

Mode relay
BindAddress 0.0.0.0
ORPort 443
IsBridge 1
AdvertiseAddress YOUR_PUBLIC_IP
DataDirectory /var/lib/moor-bridge
Verbose 1

Run with moor -f moorrc. Use port 443 to blend with HTTPS. On startup, the bridge logs its bridge line:

bridge line: shade 203.0.113.5:443 a1b2c3d4e5f6...64_hex_chars

Share this line with clients out-of-band.

2. Connect via Bridge

Add the bridge line to your client moorrc:

SocksPort 9050
UseBridges 1
Bridge shade 203.0.113.5:443 a1b2c3d4e5f6...64_hex_chars

The three parts of a bridge line:

  • Transport — scramble, shade, or mirage
  • Address — bridge IP and port (e.g. 203.0.113.5:443)
  • Fingerprint — 64-char hex Ed25519 identity public key

The fingerprint authenticates the bridge — without it, a MITM could impersonate it.

Pluggable Transports

Pluggable transports disguise MOOR traffic to defeat deep packet inspection:

shade — Elligator2-style key obfuscation + AEAD framing. Full DPI resistance.

mirage — fake TLS 1.3 handshake with real x25519 DH. Looks like normal HTTPS.

scramble — random-byte stream with AEAD framing + probe resistance. Default.

BridgeDB Distribution

BridgeDB distributes bridge lines to users via HTTPS. It rate-limits requests and partitions bridges across distribution strategies to resist enumeration.
