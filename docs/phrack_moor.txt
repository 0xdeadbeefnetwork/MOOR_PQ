--[ 0 - Why I Wrote This

I built a post-quantum onion router from scratch.  30,000 lines
of C.  One developer.  No corporate backing.  No funding.  Just
stubbornness and an unreasonable amount of coffee.

This paper is not a product announcement.  I wrote it because
the engineering problems I hit -- grafting lattice-based KEM onto
a Noise handshake, fragmenting 1088-byte ciphertexts across
fixed-size cells, making PIR work over a real DHT, getting
Elligator2 key encoding to survive DPI in the wild -- are
problems that anyone building post-quantum transport will
eventually face.  The solutions and the failures are worth
documenting.

I also wrote it because I want you to break it.  Section 8 is a
roadmap.  The code is MIT licensed and sitting on GitHub.  If
something in here is wrong, I need to know before more people
depend on it.


--[ 1 - The Problem Nobody Wants to Solve

The harvest-now-decrypt-later threat is real.  NIST standardized
ML-KEM (Kyber) and ML-DSA (Dilithium) in August 2024 [1][2]
specifically because this threat is considered imminent enough to
act on.  Every day that key exchange traffic flows over
classical-only DH is another day of recordings in somebody's
data lake.

Tor added an experimental Kyber handshake in 2024 (Proposal 332)
for the link layer.  Circuit keys are still classical.  Hidden
service rendezvous is still classical.  Consensus signatures are
still classical.  And the codebase is 800K+ lines of C with 20
years of accumulated abstractions that make a full PQ transition
a multi-year project.

That's not a criticism.  Tor has a 20-year track record of
keeping people alive.  But someone has to explore what a
clean-room PQ anonymous network looks like when you design it
from day one with lattice-based crypto as a hard requirement.

So I wrote one.


--[ 2 - What MOOR Is (Briefly)

MOOR is a 3-hop onion routing network.  Single binary: client,
relay, directory authority, hidden service, bridge, BridgeDB,
OnionBalance.  Dependencies: libsodium, zlib, pthreads.  No
OpenSSL.

Current network: 2 directory authorities, 4 relay nodes (2 in
the US, 2 in the Netherlands).  All PQ-capable.

Every key exchange is hybrid X25519 + Kyber768.  Every consensus
signature is Ed25519 + ML-DSA-65.  There is no classical-only
mode.

The code is at github.com/0xdeadbeefnetwork/MOOR_PQ.  The
network dashboard is at moor.afflicted.sh.  I'll keep the
self-promotion to this paragraph.


--[ 3 - Grafting Kyber768 onto Noise_IK Without Killing It

----[ 3.1 - The Link Handshake

Every MOOR connection uses Noise_IK [3]:

  Noise_IK_25519_ChaChaPoly_BLAKE2b

IK means the initiator knows the responder's static key (from
the consensus).  1-RTT, mutually authenticated, forward-secret.

  Message 1 (80 bytes):
    [eph_pk:32][encrypted_static:32+16_mac]

  Message 2 (48 bytes):
    [eph_pk:32][encrypted_empty:0+16_mac]

  DH ops:
    es = X25519(e_init, s_resp)
    ss = X25519(s_init, s_resp)
    ee = X25519(e_init, e_resp)
    se = X25519(s_init, e_resp)

  Transport keys:
    k_send, k_recv = HKDF-BLAKE2b(chaining_key, empty)

This is ~300 lines of C.  It gives us everything TLS gives
except the CA ecosystem, which we don't want anyway.  The relay's
Ed25519 identity key IS the trust anchor.


----[ 3.2 - The Post-Handshake PQ Upgrade

Here is where it gets interesting.  Kyber768 public keys are
1184 bytes and ciphertexts are 1088 bytes.  You cannot cram
that into a Noise handshake message without breaking the Noise
framework's assumptions about message sizes and patterns.

The solution: don't modify Noise at all.  Complete the classical
Noise_IK handshake, then upgrade the session keys.

  Client:  kem_pk, kem_sk = KEM.KeyGen()
           send kem_pk (1184 bytes) over encrypted link

  Relay:   kem_ct, kem_ss = KEM.Enc(kem_pk)
           send kem_ct (1088 bytes) over encrypted link

  Client:  kem_ss = KEM.Dec(kem_ct, kem_sk)

  Both:    new_send = HKDF(old_send_key, kem_ss)
           new_recv = HKDF(old_recv_key, kem_ss)

The key insight: the Noise session is already encrypted and
authenticated.  The KEM exchange happens inside that tunnel.
An attacker who records the handshake and later breaks X25519
can recover the Noise session keys, but then faces 1184 bytes
of Kyber768 public key and 1088 bytes of ciphertext that they
still can't break.

The HKDF mixing is critical.  If you just replaced the keys,
a broken Kyber768 would leave you with nothing.  By mixing
KEM shared secret into existing session keys, you get hybrid
security: break BOTH or break nothing.


----[ 3.3 - What Can Go Wrong

Timing.  The KEM.KeyGen() call takes ~150us on modern hardware,
KEM.Enc() about the same.  This is 100x slower than X25519.
If a relay is handling 1000 simultaneous connections, that's
150ms of CPU time just for KEM operations.  Currently not a
problem at 4 relays with low traffic, but it will be.

The 1184-byte public key and 1088-byte ciphertext travel as raw
bytes over the encrypted link after the Noise handshake.  This
means the link-layer PQ upgrade is synchronous -- the connection
blocks until both sides complete the exchange.  For a non-
blocking event loop, this is ugly.  I handle it by doing the PQ
upgrade in the connection setup phase before the connection is
registered in the event loop.  It works, but it means connection
establishment is ~3ms instead of ~1ms.

If Kyber768 is broken but the attacker doesn't know it yet, the
HKDF mixing still provides classical Noise_IK security.  But if
X25519 is broken (quantum) and Kyber768 is somehow also broken,
you have nothing.  The hybrid assumption is that breaking both
simultaneously is harder than breaking either one.  This is
widely believed but not formally proven for arbitrary adversary
models.


--[ 4 - Circuit Key Exchange: Making Hybrid Actually Work

----[ 4.1 - CKE Design

Circuit key exchange (CKE) is a custom handshake loosely
inspired by Tor's ntor but using the relay's Ed25519 identity
key (converted to Curve25519) instead of a separate onion key.

  Client -> Relay:  CREATE_PQ [relay_id_pk:32][eph_pk:32]
  Relay -> Client:  CREATED_PQ [relay_eph_pk:32][auth_tag:32]

  dh1 = X25519(client_eph, relay_eph)     forward secrecy
  dh2 = X25519(client_eph, relay_curve)   identity binding

  salt = BLAKE2b("moor-cke-v1" || relay_id_pk)
  ck, key_seed = HKDF(salt, dh1 || dh2)
  auth = BLAKE2b-MAC(ck, "moor-cke-verify" || pks)

  Verify: auth_tag == auth

Then the KEM phase:

  Client:  kem_ct, kem_ss = KEM.Enc(relay_kem_pk)
           send CELL_KEM_CT cells

  Relay:   reassemble kem_ct from CELL_KEM_CT cells
           kem_ss = KEM.Dec(kem_ct, kem_sk)

  Both:    hybrid = BLAKE2b(key_seed || kem_ss)
           fwd_key = KDF(hybrid, 1, "moorFWD!")
           bwd_key = KDF(hybrid, 2, "moorBWD!")

24 lines of C in moor_crypto_circuit_kx_hybrid().


----[ 4.2 - The 1088-Byte Problem

Kyber768 ciphertexts are 1088 bytes.  MOOR cells carry 509
bytes of payload.  You can't fit a KEM ciphertext in one cell.

Options I considered:

  a) Send raw bytes over the link (original approach).  Works
     for the first hop because the link is already encrypted.
     Fails for EXTEND to middle/exit because you're inside an
     onion circuit and everything must be cells.

  b) Fragment into 3 cells (509 + 509 + 70 bytes).  Adds a
     reassembly state machine.  The relay must buffer partial
     KEM ciphertexts, which creates a DoS vector: send 2 of 3
     fragments, hold state, repeat 10000 times.

  c) New cell type CELL_KEM_CT (command byte 8) carrying
     fragments with the circuit_id for demultiplexing.

I went with (c).  The relay side reassembles into the circuit's
pq_kem_ct[] buffer (1088 bytes pre-allocated per circuit).
pq_kem_ct_len tracks bytes received.  pq_kem_pending flags that
we're waiting for the full ciphertext before processing the
CREATE_PQ.

The DoS vector from (b) still exists -- you can send partial
KEM_CT cells and hold circuit state.  Mitigation: a timeout.
If the full 1088 bytes don't arrive within 10 seconds, the
circuit is torn down.  This is not perfect.  An attacker who
controls a guard can hold 1000 circuits in the KEM-pending
state using 1088 * 1000 = ~1MB of relay memory.  Not great,
not catastrophic.


----[ 4.3 - Attacking the Handshake

The auth_tag in CREATED_PQ proves the relay controls its
identity secret key.  Without it, a MITM could substitute their
own ephemeral key.  But the auth_tag is computed over the
classical DH values only -- it doesn't cover the KEM exchange.

This means: if an attacker can break Kyber768 (not currently
feasible), they can substitute their own KEM ciphertext after
the CREATED_PQ auth check passes.  The classical DH portion
would still be intact, so the attacker gets nothing unless they
ALSO break X25519.  But it's an inelegance.  A cleaner design
would include the KEM shared secret in the auth tag computation.

I haven't fixed this yet because it requires the relay to send
auth_tag AFTER the KEM exchange completes, which changes the
message flow.  It's on the list.

Another attack: an active MITM who controls the network between
client and guard can delay CELL_KEM_CT cells selectively to
create a timing side channel.  If the KEM processing time is
observable (it is -- ~150us), the MITM can distinguish between
KEM.Dec(valid_ct) and KEM.Dec(garbled_ct).  This leaks whether
the client has the correct relay KEM public key, which leaks
which relay the client is connecting to.  Constant-time KEM
decapsulation mitigates this, and Kyber768 reference code is
constant-time, but the surrounding C code (buffer copies,
conditional branches on pq_kem_ct_len) is not formally verified
for timing leaks.


--[ 5 - Everything Looks the Same: Traffic Shaping

----[ 5.1 - 532 Bytes, Always

Every cell on the wire is 532 bytes:

  [length:2][ciphertext:514][poly1305_mac:16]

The length field is always 530 (514 + 16).  The ciphertext is
ChaCha20-Poly1305 IETF AEAD.  IND-CPA secure.

No variable-length records.  No protocol headers.  No magic
bytes.  A 10-byte ping and a 498-byte download produce
identical 532-byte packets.  This is the first line of defense
against traffic fingerprinting.


----[ 5.2 - WTF-PAD: When Adaptive Padding Isn't Enough

WTF-PAD (Juarez et al., ESORICS 2016 [4]) uses probabilistic
state machines to inject dummy packets.  MOOR implements 3
built-in machines.  I'll show the web machine:

  State 0 (IDLE):
    IAT:  400-600ms, 50% pad probability
  State 1 (BURST):
    IAT:  5-15ms (70%), 15-30ms (30%); 80% self-loop
  State 2 (GAP):
    IAT:  40-60ms, transition to BURST on real cell

Each circuit gets a randomized copy:

  - IAT bin boundaries: +/- 30%
  - IAT probabilities: +/- 25%, renormalized
  - Transition probabilities: +/- 20%
  - Max padding cells: +/- 50%

No two circuits produce the same traffic pattern.

But here's the problem.  Sirinam et al. (CCS 2018 [5]) showed
that deep learning classifiers defeat WTF-PAD with >98%
accuracy on a closed-world dataset.  The CNN doesn't care about
individual packet timing -- it learns burst patterns, directional
sequences, and volume distributions that WTF-PAD doesn't mask.

WTF-PAD is necessary but insufficient.


----[ 5.3 - Layered Defense and Why You Need All Three

MOOR's traffic analysis defense is three layers:

  Layer 1: WTF-PAD (adaptive, masks burst patterns)
  Layer 2: Constant-rate floor (50 cells/sec, masks activity)
  Layer 3: Poisson relay mixing (decorrelates input/output)

The constant-rate padding is trivial: one CELL_PADDING every
20ms regardless of real traffic.  This makes it impossible to
determine if a circuit is active or idle by counting cells.

Poisson mixing is more interesting.  At each relay, cells enter
a mixing pool and are released after an exponential random
delay:

  delay = -ln(u) * lambda_ms,  u ~ Uniform(0,1]

Real traffic and padding share the pool.  An observer watching
a relay's input and output cannot correlate which incoming cell
became which outgoing cell because the release order is
randomized.  Per-circuit FIFO is preserved (cells within the
same circuit are never reordered).

The combination addresses Sirinam's attack: WTF-PAD masks
packet-level timing, constant-rate masks activity/idle
transitions, and mixing decorrelates flow-level patterns.

Does this actually work?  I haven't done a formal evaluation
against the Sirinam classifier.  That's research I intend to
do but haven't.  I'm being honest about this because claiming
traffic analysis resistance without measurement is worse than
claiming nothing.

What I can say: the bandwidth overhead is ~40% for constant-rate
padding at 50 cells/sec, which is substantial.  If you're
running a relay on a metered connection, this costs money.
There's a config toggle.  Some users will turn it off.  Those
users will be fingerprintable by the absence of padding.  This
is the oldest problem in anonymity networks and I don't have a
clever solution for it.


--[ 6 - Private Information Retrieval for Hidden Services

----[ 6.1 - The Metadata Leak Nobody Talks About

When you visit a .onion address on Tor, you fetch the hidden
service descriptor from an HSDir relay.  That relay learns
exactly which .onion address you requested.  If the relay is
operated by an adversary, they now know you're interested in
that specific hidden service.

Tor has known about this for years.  The defense is to use a
fresh circuit for the descriptor fetch so the HSDir doesn't
know your real IP.  But it still knows which descriptor you
wanted, which is itself a significant metadata leak.


----[ 6.2 - XOR-PIR Implementation

MOOR uses computational PIR (Chor et al., FOCS 1995 [6];
Mittal et al., USENIX Security 2011 [7]):

  1.  Client generates random 32-byte bitmask S1
  2.  Compute S2 = S1 XOR target_query
  3.  Send S1 to DHT replica 0
  4.  Send S2 to DHT replica 1
  5.  R0 = XOR of descriptors selected by S1
  6.  R1 = XOR of descriptors selected by S2
  7.  Result = R0 XOR R1

Neither replica learns which descriptor was requested.  Each
sees only a random bitmask.

Wire format:

  Query:    [query_id:4][bitmask:32] = 36 bytes
  Response: [query_id:4][total_len:2][offset:2][data:<=490]


----[ 6.3 - Breaking PIR: Collusion and Timing

The obvious attack: if both replicas collude, they can XOR
S1 and S2 to recover the target query.  S1 XOR S2 = target.
Game over.

In a 4-relay network, the probability that both DHT replicas
for a given descriptor are controlled by the adversary is high.
PIR provides meaningful protection only when the network is
large enough that colluding replicas are unlikely.  With 4
relays, PIR is security theater.  With 100+ relays, it becomes
genuinely useful.  I ship it now so it works when the network
grows.

Timing attack: the client sends PIR queries to two different
relays at roughly the same time.  A network-level observer can
correlate these two connections and infer that they're part of
the same PIR query.  This doesn't reveal the target descriptor
directly, but it narrows the anonymity set.  Mitigation: add
random delay between the two queries.  Current implementation
doesn't do this.  It should.

Resource exhaustion: a PIR response requires the relay to XOR
every descriptor in its store against the bitmask.  With 10,000
descriptors at ~1KB each, that's 10MB of XOR per query.  An
attacker can flood PIR queries and consume relay CPU.  The
token bucket rate limiter helps, but a determined attacker with
many circuits can still degrade service.


--[ 7 - Making DPI Useless: Pluggable Transports

MOOR ships 3 transports compiled into the binary.  No external
processes.


----[ 7.1 - Scramble and Probe Resistance

Scramble is the default.  Handshake:

  Client: [eph_pk:32][random_pad:32-480][HMAC:32]
  Server: [eph_pk:32][random_pad:32-480][HMAC:32]

Variable-length random padding defeats active probing: a censor
cannot send a fixed-size probe and check the response.  The HMAC
is keyed with DH(client_eph, bridge_identity).  Without the
bridge's identity key, an active prober gets no valid response
-- the bridge ignores the connection.

Post-handshake: frame lengths are XORed with ChaCha20 keystream
from a header key.  Even frame boundaries are opaque.

How to detect Scramble: entropy analysis.  Scramble traffic is
indistinguishable from random bytes, but MOST protocols are NOT
random bytes.  If a censor blocks all connections whose first
N bytes have Shannon entropy > 7.9 bits/byte, they'll catch
Scramble (and everything else that uses encryption without a
cleartext header).  China's GFW already does this to some
extent.  This is why Mirage exists.


----[ 7.2 - Shade: Elligator2 in Practice

Standard X25519 public keys have detectable structure: not every
32-byte string is a valid Curve25519 point.  A DPI system can
check this.  Elligator2 (Bernstein et al., CCS 2013 [8]) maps
Curve25519 points to uniform random strings.

Shade uses Elligator2 for key encoding, BLAKE2b HMAC for the
mark (which binds the handshake to the bridge identity without
revealing it to a passive observer), and ChaCha20-Poly1305 AEAD
for transport.

IAT mode 2 adds random fragmentation and random inter-packet
delay, breaking the fixed 532-byte cell pattern at the transport
layer.

How to detect Shade: the Elligator2 encoding is not perfectly
uniform.  Bernstein et al. note that only ~50% of Curve25519
points have an Elligator2 representative.  The keygen must
reject points without representatives and retry.  If the
implementation leaks timing information during this
rejection-sampling loop, a network observer can measure
connection setup time and distinguish Shade from random.
The current implementation does retry in a loop, and the number
of iterations is not constant.  This is a real fingerprint.


----[ 7.3 - Mirage: Faking TLS 1.3 (and How to Detect It)

Mirage wraps MOOR cells in TLS 1.3 records.

  ClientHello:
    SNI: random domain from 64 hardcoded options
    cipher_suites: 0x1301, 0x1302, 0x1303
    key_share: real X25519 ephemeral

  ServerHello:
    key_share: real X25519 ephemeral

  Key derivation:
    shared = X25519(our_sk, their_pk)
    transcript = BLAKE2b(client_hello || server_hello)
    keys = BLAKE2b-keyed(shared || transcript,
                         "moor-tls-camo-keys")

Post-handshake: cells wrapped in TLS Application Data records
(content type 23) with variable lengths.

How to detect Mirage:

  1.  Key derivation uses BLAKE2b, not HKDF-SHA256.  A censor
      who MITMs the handshake and tries to derive keys with
      standard TLS 1.3 KDF will fail.  This requires active
      interception, which has political/legal costs.

  2.  The 64 hardcoded SNI domains.  A censor who compiles a
      list of these domains and blocks connections with matching
      SNIs defeats Mirage trivially.  The domain list should be
      configurable and per-client, not hardcoded.  It isn't yet.

  3.  JA3/JA4 fingerprinting.  The ClientHello includes a
      specific set of extensions, cipher suites, and supported
      groups.  If this fingerprint doesn't match any known
      browser, it's detectable.  Current Mirage makes no effort
      to mimic a specific browser's JA3 hash.

  4.  Certificate expectation.  Real TLS 1.3 includes a server
      certificate.  Mirage skips this -- after ServerHello, it
      goes straight to application data.  Any censor that checks
      for the encrypted_extensions/certificate/certificate_verify
      handshake messages will notice they're missing.

Mirage is the weakest transport against a sophisticated censor.
It's effective against naive DPI that just checks the content
type byte, but it will not survive a well-funded adversary.
Scramble and Shade are stronger against entropy-agnostic
analysis; Mirage is stronger against entropy-based blocking.
Pick your adversary.


--[ 8 - How to Attack MOOR

This is the section I want you to read.


----[ 8.1 - The 2-DA Problem

MOOR has 2 directory authorities.  Both are operated by me.
If I'm compromised, an attacker can forge the consensus and
inject malicious relays.  Game over.

This is the single biggest weakness in the current deployment.
The protocol supports N-of-M voting with dual Ed25519 + ML-DSA-65
signatures, but the implementation currently requires both DAs
to agree.  With 2 DAs, "both" means "all."  Production needs
5-9 DAs operated by independent parties.  I don't have 5-9
independent parties.


----[ 8.2 - Path Selection with 4 Relays

4 relays.  1 guard (DROPOUT), 1 middle-capable that isn't guard
or exit (TURBINE), 1 exit (VALIDATOR), 1 middle (RAZORGIRL).
The bandwidth-weighted path selection algorithm exists, the
GeoIP diversity constraints exist, the vanguard rotation exists.
None of it matters when there's only one possible path.

Every circuit goes: DROPOUT -> TURBINE or RAZORGIRL -> VALIDATOR.
The anonymity set is 1.  An adversary who controls any relay in
the path can trivially correlate traffic.

This improves linearly with relay count but isn't meaningful
until ~50 relays across diverse jurisdictions.


----[ 8.3 - Memory Safety in 30K Lines of C

I wrote 30,000 lines of C.  Alone.  With no formal review.

The code compiles with -Wall -Wextra -Werror -fstack-protector-
strong -D_FORTIFY_SOURCE=2 -pie -Wl,-z,relro,-z,now.  There are
bounds checks on buffer operations.  libsodium's constant-time
primitives are used for all crypto.  But this is still C.  There
are certainly memory bugs I haven't found.

The vendored Kyber768 and Dilithium3 implementations are NIST
reference code.  Reference code is written for clarity, not
hardening.  It has not been independently audited for this
integration.  Buffer overflows in KEM decapsulation or signature
verification would be catastrophic.

If you're going to audit one thing: look at the relay cell
processing in relay.c.  It handles untrusted input from the
network, parses variable-length payloads, and touches multiple
shared data structures.  Lines 1800-3400.


----[ 8.4 - Timing Correlation

A global passive adversary who can observe both the client's
connection to the guard and the exit's connection to the
destination can correlate timing and break anonymity.  This is
true of every low-latency anonymity network.  MOOR's constant-
rate padding and Poisson mixing raise the cost but do not
eliminate the attack.

More practically: MOOR's guard selection pins to a single guard
for 120 days (Tor-aligned).  If the guard is compromised, the
attacker can observe all traffic entering the network from that
client for 4 months.  Combined with timing at the exit, this is
likely sufficient for deanonymization.

Vanguard relays mitigate guard discovery for hidden services
specifically, but client-side guard pinning is a fundamental
tension between performance (stable paths) and security
(rotating entry points).


----[ 8.5 - Vendored Crypto

The Kyber768 and Dilithium3 implementations are vendored from
the NIST PQC submission reference code.  They are not
libsodium.  They have not had the same level of scrutiny.

Specific concerns:

  - Stack allocation of large arrays (Kyber768 has internal
    arrays up to 2400 bytes).  Stack overflow on constrained
    relays.
  - Reference code does not zero sensitive memory on all paths.
    I added moor_crypto_wipe() calls, but I may have missed
    some.
  - No formal verification that the implementation matches the
    spec.  If there's a subtle bug in polynomial arithmetic,
    the KEM might be weaker than Kyber768's claimed security
    level.

If NIST or a team like PQCRYPTO publishes hardened C
implementations, MOOR should switch.  Until then, this is what
we have.


----[ 8.6 - Side Channels in Non-Constant-Time Code

libsodium primitives are constant-time.  But the code AROUND
them is not.

  - pq_kem_ct_len is checked with a branch before calling
    KEM.Dec().  This leaks whether a full ciphertext has been
    received.
  - The WTF-PAD state machine uses floating-point probability
    comparisons.  Floating-point timing varies by value on some
    architectures.
  - Circuit hash table lookup is open-addressing with linear
    probing.  Timing depends on occupancy and probe count.
  - The PIR bitmask XOR loop iterates over all stored
    descriptors.  If some descriptors are in cache and others
    aren't, memory access patterns leak information.

None of these are likely exploitable over the network due to
noise, but a co-located attacker on the same physical machine
(cloud VM neighbor) might be able to measure them.


--[ 9 - Lessons from Writing 30K Lines of Crypto C Solo

  1.  The hard part isn't the crypto.  The hard part is the
      state management.  Circuit builds fail halfway.  TCP
      connections drop during KEM exchange.  The builder thread
      races with the event loop.  The elegant handshake protocol
      meets the ugly reality of non-blocking I/O.

  2.  Blocking code is easier to reason about.  Non-blocking
      code is easier to scale.  Pick one, pay the price.  I
      started with blocking (builder thread), hit a wall at 40
      connections to the same guard, and rewrote to a non-
      blocking state machine.  Should have done it first.

  3.  Fixed-size cells are a gift.  No buffer management, no
      fragmentation (except for KEM ciphertexts), no MTU
      negotiation.  514 bytes in, 532 bytes out, always.

  4.  Test on real networks early.  The code worked perfectly
      on localhost and broke immediately when relays were on
      different continents with 200ms RTT.  Timeouts, retries,
      and congestion control only matter when latency is real.

  5.  Write the security analysis before the code.  I did it
      the other way around and found design flaws (like the
      auth_tag not covering the KEM exchange) only after
      implementation.

  6.  Vendoring crypto is terrifying.  Every line of Kyber768
      reference code that touches polynomial arithmetic is a
      line I cannot verify.  But the alternative -- no PQ
      crypto -- is worse.

  7.  Nobody will run your relay unless it's one command to
      install.  ./configure && make && sudo make install.  If
      it needs a Docker compose file with 6 services, it's
      dead on arrival.
